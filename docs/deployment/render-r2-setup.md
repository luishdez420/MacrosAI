# Render And Cloudflare R2 Setup

Last updated: 2026-07-21

Related: [[release-runbook|Release And Beta Runbook]], [[edge-security-runbook|Edge Security Runbook]], [[../Architecture|Architecture]], [[../Known Issues|Known Issues]].

The selected production topology is Render for the API, workers, PostgreSQL, and Key Value store, plus Cloudflare R2 for private meal images. The repository's `render.production.yaml` creates the paid Render resources and references managed values; it contains no production credentials. The root `render.yaml` is intentionally a no-cost personal preview and is documented separately in [[render-free-preview-setup|Free Render Preview Setup]].

## Before Creating The Blueprint

1. Create separate Render preview and production environments. Keep their database instances, Key Value instances, R2 buckets, Clerk tenants, audit receivers, and Sentry environments separate.
2. Create a private Cloudflare R2 bucket for each environment. Do not enable a public bucket or a public custom domain for meal images.
3. Create an R2 S3 API token scoped to **Object Read & Write** for only that environment's bucket. Record its Access Key ID and Secret Access Key in Render managed environment settings. The secret is displayed once by Cloudflare; never add it to a `.env` file committed to Git.
4. Record the R2 S3 endpoint as `https://<CLOUDFLARE_ACCOUNT_ID>.r2.cloudflarestorage.com`, with region `auto`.
5. Provision Clerk, Sentry, USDA/OpenAI, and the approved HTTPS audit receiver before enabling a production API deployment.

Cloudflare R2 automatically encrypts stored objects at rest. It does not support standard AWS S3 server-side-encryption request headers, so `IMAGE_STORAGE_S3_COMPATIBILITY=cloudflare_r2` instructs the backend to omit those headers while preserving the existing private-prefix and short-lived presigned-URL controls.

## Create Render Resources

1. Connect the Git repository in Render and create a Blueprint from `render.production.yaml`.
   After authenticating the official Render CLI and selecting the intended workspace, run
   `render blueprints validate render.production.yaml` before applying it. The CLI performs schema, semantic,
   and existing-resource conflict validation that local YAML parsing cannot provide.
2. Select the intended region before the first deployment. The Blueprint uses `oregon` by default; change every Render resource together if a different region is required.
3. Keep the `living-nutrition-postgres` and `living-nutrition-redis` public allowlists empty. Render private-network service connections continue to work while public datastore connections are blocked.
4. On the API service, provide every `sync: false` value in Render's Environment page. The workers obtain the same values by service reference and must not be given independent secrets.
5. Do not run migrations manually from multiple worker shells. The API pre-deploy command owns the reviewed migration; worker commands retain a safe `alembic upgrade head` guard for a new environment or a redeploy race.

## Required API Values

Set these values only in the Render dashboard or a managed Render environment group:

| Variable | Value or source |
| --- | --- |
| `CLERK_JWKS_URL`, `CLERK_ISSUER`, `CLERK_AUDIENCE` | Production Clerk tenant values |
| `ADMIN_CLERK_SUBJECTS` | Comma-separated authorized operational-review Clerk subject IDs |
| `OPENAI_API_KEY`, `USDA_API_KEY` | Backend-only provider keys |
| `SENTRY_DSN` | HTTPS backend DSN for the production Sentry environment |
| `AUDIT_LOG_RETENTION_DAYS` | Approved retention duration |
| `AUDIT_DELIVERY_WEBHOOK_URL`, `AUDIT_DELIVERY_HMAC_SECRET` | Approved HTTPS append-only receiver and 32+ character secret |
| `IMAGE_STORAGE_S3_BUCKET` | Private environment-specific R2 bucket |
| `IMAGE_STORAGE_S3_ENDPOINT_URL` | Account-specific HTTPS R2 S3 endpoint |
| `IMAGE_STORAGE_S3_ACCESS_KEY_ID`, `IMAGE_STORAGE_S3_SECRET_ACCESS_KEY` | Bucket-scoped R2 Object Read & Write credentials |
| `TRUSTED_PROXY_CIDRS` | Only confirmed direct ingress proxy CIDRs |

`JWT_SECRET` and `METRICS_BEARER_TOKEN` are generated by the Blueprint. Never copy either value to the mobile client. Mobile builds receive only `EXPO_PUBLIC_API_URL`, Clerk's publishable key, and, when enabled, the public mobile Sentry DSN/environment through EAS secrets.

## Trusted Proxy Requirement

The API intentionally refuses production startup without `TRUSTED_PROXY_CIDRS`. Do not use `0.0.0.0/0` or Render outbound-IP ranges: neither proves that a direct inbound peer is trusted to supply client-address headers. Confirm the ingress proxy ranges for the chosen Render configuration with Render's current platform guidance or support, then record the validation date in the release record. If the platform cannot provide a safe stable allowlist, introduce a separate approved ingress layer with documented proxy ranges before production release.

## Validation

1. Confirm the API is healthy at `/health`, then confirm `/api/v1/health/ready` reports healthy database, Redis rate limiter, and provider circuit dependencies.
2. Confirm the PostgreSQL and Key Value stores reject public access while the Render services work through their private connection strings.
3. Upload a consented preview image through camera analysis, confirm the object is in the private R2 bucket under the configured prefix, and verify only an owner-authorized URL can access it.
4. Confirm a URL expires within the configured 15-minute maximum and that image deletion, retention expiry, and account deletion remove the R2 object.
5. Trigger one safe preview error and verify the Sentry event has only approved correlation tags.
6. Follow [[release-runbook|the release and beta runbook]] for physical-device and rollback checks.
