### 2026-07-21 privacy lifecycle update

This update supersedes older references below to a “future” completed-meal-photo preference. Camera scans are temporary by default. After a successful analysis, normalized inputs remain only through the bounded review window; failed or cancelled jobs delete them immediately. During meal confirmation, a user may explicitly choose to keep those scans. The API copies the images into private owner-scoped meal storage, applies the user's saved retention duration, returns only safe image metadata with the meal, and deletes temporary inputs. Saved Meal Detail can delete a retained image, while a separate owner-authorized endpoint can create a short-lived private-storage read URL. Account deletion still removes all remaining private assets first or fails safely for retry. Local private storage cannot create a network read URL, so actual image viewing requires configured S3-compatible storage.

## Known technical constraints

- Camera-based nutrition remains an estimate because a photo cannot reliably reveal exact weight, cooking oil, sauces, hidden ingredients, edible portion, or cooking method.
- Clerk manages mobile sign-in, verification, recovery, configured OAuth, and sessions when `EXPO_PUBLIC_CLERK_PUBLISHABLE_KEY` is configured. FastAPI verifies Clerk JWTs through JWKS and maps the verified subject to an internal user. Local password/JWT sessions remain for non-production preview compatibility and a bounded migration path only. Credential and image-analysis endpoints use configurable in-memory limits in development and atomic Redis rolling windows in production. Production startup requires Clerk mode, Clerk JWKS/issuer settings, a strong `JWT_SECRET`, disabled compatibility modes, and an enabled Redis limiter; tenant configuration, trusted-proxy policy, production deployment validation, and richer session policy remain incomplete.
- Sentry reporting is optional outside production and does not initialize in Expo Go without a public mobile DSN. Production API startup requires an HTTPS `SENTRY_DSN`. The repository has no Sentry project configuration, alert policy, source-map/debug-symbol upload, or release-artifact validation yet; deployers must verify the sanitized-event contract before relying on it for incident response.
- External nutrition provider search is still attempted for stale, expired, or uncached active search flows. Barcode lookup checks user-created barcode products and cached Open Food Facts source records before external providers; provider search/detail lookups seed normalized source records, and complete fresh query result sets use a bounded query-to-source-record cache before provider calls. A transient HTTP failure at one provider no longer prevents a healthy fallback provider from responding. A provider circuit opens after repeated transient failures and allows a bounded half-open recovery probe; production configuration requires its shared Redis state, while local preview uses the in-memory implementation. Stale detail and exact barcode refreshes claim a database-backed lease, then use jittered exponential backoff after a provider failure or no match while returning the flagged cached snapshot. The separate refresh worker periodically considers a bounded batch of eligible stale source records using that same coordination, but it deliberately does not prefetch arbitrary queries. Search can fall back to cached matches after a provider failure; a total live-provider outage returns a safe `503` with a request ID. USDA and Open Food Facts requests use configurable timeouts and bounded retries for transient failures. Protected process metrics expose bounded provider outcome/latency/circuit and cache hit/refresh/fallback signals, but dashboard/alert operation and real deployed outage validation are still pending.
- SQLite phone preview uses SQLAlchemy metadata bootstrap instead of the PostgreSQL Alembic chain. The initial historical Alembic revision contains PostgreSQL-specific `JSONB`, so running `alembic upgrade` against SQLite is unsupported; production migrations must be validated against PostgreSQL.
- Expo Go phone testing depends on LAN reachability between the phone and Mac.
- Development-client phone testing also depends on an EAS project link, Apple/Android signing setup, installation of the generated internal build, and LAN reachability to the local API. The repository does not include an EAS project ID or credentials.
